# 경쟁사 분석 (2026-07 조사)

문제 정의([../Problem/문제-정의.md](../Problem/문제-정의.md))를 기준으로 경쟁 환경을 조사. 웹 조사 기반이며 출처는 하단에 표기.

> 후속 문서: 경쟁 요소별 제공 수준을 점수화한 [전략-캔버스.md](전략-캔버스.md) (Blue Ocean 전략 캔버스 + ERRC).

## 분류 기준

- **직접 경쟁자** — 나와 같은 문제를 정의하고, **솔루션의 형태까지 동일한** 회사
- **간접 경쟁자** — 나와 **같은 문제를 정의**하지만 다른 방식으로 해결하는 회사
- **Status Quo** — 이런 솔루션을 안 쓸 때 고객이 취하는 행동 (로그 뒤지기, 스크립트 짜기 등)

> ⚠️ 전제: 우리 솔루션은 아직 확정되지 않았다. 아래 분류는 문제 정의에서 유도한 가정 — "에이전트의 자율적 툴 콜/API 호출을 **실행 전에(request-path)** 추적·통제하는 제품" — 을 기준으로 한 것이다. 솔루션 방향이 바뀌면 직접/간접 경계도 다시 그려야 한다.

---

## 1. 직접 경쟁자

### SatGate — 가장 직접적인 경쟁자 ★

"The Economic Firewall for AI Agent Requests." 문제 정의가 우리와 거의 동일하다:

> "Standing API keys and after-the-fact alerts are a bad fit for autonomous systems" — 정적 API 키와 사후 알림으로는 자율 에이전트를 통제할 수 없다.

| 항목 | 내용 |
|---|---|
| 솔루션 형태 | API 앞단에 놓는 게이트웨이(Go 프록시). 표준 게이트웨이 / 사이드카 / **MCP 프록시** 3가지 배포 모드 |
| 핵심 기능 | 요청 실행 **전** 예산 집행(request-path budget enforcement), 에이전트별 예산·비용 귀속, macaroon 기반 위임(에이전트 A가 B에게 $50 한도의 하위 토큰 발급, 권한 상승 불가), MCP JSON-RPC 페이로드를 읽어 **툴 콜 단위로 비용 귀속**, 서명된 감사 영수증(Policy-to-Proof) |
| 배포/가격 | 오픈소스 + 매니지드 클라우드(cloud.satgate.io). 무료 티어 + 엔터프라이즈 플랜(가격 미공개) |
| 성숙도 | **초기(프리론칭)**. 디자인 파트너 모집 중, 특허 출원 중 |
| 타겟 | 자율 에이전트를 배포하는 엔터프라이즈·플랫폼 운영사 |

**판정: 직접 경쟁자 맞음.** 문제 정의(사후 추적으로는 부족, 실행 전 통제 필요)와 솔루션 형태(호출 경로의 게이트웨이)가 모두 겹친다. 다만 프리론칭 단계라 시장을 선점한 상태는 아니다 — 오히려 이 문제에 베팅하는 팀이 또 있다는 문제 검증 신호로 읽어야 한다.

### Cycles (runcycles/cycles-protocol) — 직접 (조건부)

AI 에이전트의 런타임 예산·행동 권한 통제를 위한 **오픈 프로토콜**(Apache 2.0).

| 항목 | 내용 |
|---|---|
| 솔루션 형태 | 게이트웨이가 아니라 **프로토콜/SDK 명세**. 에이전트가 실행 전 예산을 예약(reserve) → 실행 → 실제 비용 커밋(commit)하는 3단계. release/extend 포함 4개 런타임 연산 |
| 핵심 기능 | 원자적 예산 예약(부분 잠금 없음), 동시성 안전(수십 개 에이전트가 공유 예산을 초과 예약 불가), 멱등 커밋(재시도 안전), 장부 일관성 보장. OpenAI·Anthropic·MCP 서버·커스텀 툴에 걸쳐 적용 |
| 성숙도 | **매우 초기**. GitHub 스타 3개, v0.1.25, 참조 구현(Python) 공개 |
| 상업성 | 오픈소스 명세. 상업 제품 아님 (runcycles가 참조 구현 운영) |

**판정: 통제 지점은 동일(실행 전 예산 집행)하지만 전달 형태가 다르다** — 제품이 아니라 프로토콜이라 개발자가 직접 서버를 구현·운영해야 한다. 경쟁자라기보다 (a) 이 프로토콜을 채택/호환하는 전략적 선택지, (b) 우리가 만들 제품이 "Cycles 호환 매니지드 서비스"가 될 수도 있는 관계. 상업적 위협은 현재 낮음.

### 기능 단위로 겹치는 플랫폼 플레이어 (직접 — 부분)

전업(pure-play)은 아니지만 동일한 기능을 이미 제공하는 게이트웨이들. 본업은 LLM 라우팅/플랫폼이고, 에이전트 비용 통제는 기능 중 하나.

| 제품 | 겹치는 기능 | 차이 |
|---|---|---|
| **LiteLLM** | 에이전트 세션별 예산 상한(max_budget_per_session)·반복 상한(max_iterations), MCP 툴 콜 라우팅 | 본업은 100+ 프로바이더 통합 게이트웨이. 거버넌스 UX·비용 귀속 리포팅은 부가 기능 수준 |
| **TrueFoundry** | 네이티브 MCP gateway, LLM + 툴 사용 통합 비용 뷰, OAuth2/RBAC | 본업은 ML/AI 인프라 플랫폼 |
| **Portkey** (Palo Alto 인수) | MCP Gateway(2026-01 GA), 팀별 비용 가시성, 가드레일 | 인수 후 보안 플랫폼으로 통합 중. 정교한 에이전틱 워크플로 지원은 제한적이라는 평가 |

## 2. 간접 경쟁자

같은 문제("에이전트/AI 지출을 추적·통제할 수 없다")를 정의하지만, 다른 방식으로 푸는 회사들.

### 관측성(Observability) — 추적하되 통제하지 않음
LangSmith, Langfuse, Helicone, Braintrust, Arize 등. 에이전트 실행 트레이스와 호출별 비용을 수집·시각화하지만, **사후 관측**이다 — 실행 전 차단·예산 집행(enforcement)은 하지 않는다. 우리 문제 정의의 "추적"만 풀고 "통제"는 안 푸는 셈. 다만 이미 트레이스 데이터를 쥐고 있어 enforcement 기능을 얹으면 직접 경쟁자로 전환될 수 있다.

### FinOps / 비용 가시성 — 사후 재무 귀속
Finout(에이전틱 AI 비용 거버넌스를 명시적으로 다룸), CloudZero, Vantage, Mavvrik(클라우드+GPU+SaaS+LLM+에이전트 워크로드 전체의 재무 귀속) 등. 같은 문제를 재무/청구 관점에서 정의하고, **빌링 데이터 기반 사후 분석**으로 푼다. 호출 경로의 실시간 통제는 없음. CFO/FinOps 팀이라는 다른 구매자를 통해 들어온다는 점에 주의 — 예산 소유자가 겹칠 수 있다.

### Agent Gateway (보안·접근 제어 중심)
Nutanix(인프라, 2026-05 GA), Arcade(인증 기반), Manufact(MCP 호스팅), Solo.io agentgateway(오픈소스, Agentic AI Foundation 기증). "에이전트의 무분별한 프로덕션 접근 통제"라는 같은 문제를 **권한·인증·보안** 관점에서 푼다. 비용은 부차적 — 단, Solo.io는 agentgateway 기반 실시간 AI 비용 통제 사례를 이미 발행했으므로 비용 영역으로의 확장이 진행 중.

### 플랫폼 내장 거버넌스
Databricks Unity AI Gateway 등. 자사 플랫폼 안의 에이전트에는 강력하지만 플랫폼 밖 트래픽은 못 본다. 하이퍼스케일러(Bedrock, Azure AI Foundry)가 유사 기능을 내장할 리스크 상존.

## 3. Status Quo — 솔루션을 안 쓸 때 고객이 하는 행동

실제 영업에서 가장 자주 마주칠 경쟁자. ⚠️ 아래는 가설이며 인터뷰로 확인 필요.

| 행동 | 구체적 모습 | 한계 (우리가 파고들 틈) |
|---|---|---|
| **로그 뒤지기** | 청구서에 이상이 보이면 프로바이더 사용량 화면·애플리케이션 로그를 수작업으로 뒤져 원인 추적 | 사후 대응. 에이전트·세션·기능 단위 귀속이 안 되고, 어떤 요청이 어떤 툴 콜 연쇄를 유발했는지 재구성에 수 시간~수 일 |
| **스크립트 짜기** | 토큰 카운팅·비용 집계 스크립트, 태깅용 래퍼 함수, 크론으로 도는 리포트를 자체 제작 | 엔지니어링 리소스 소모, 프로바이더 추가마다 유지보수. "통제(실행 전 차단)"까지 만드는 경우는 드묾 |
| **API 키 쪼개기 + 스프레드시트 정산** | 팀/프로젝트별 키 발급 후 월말에 시트로 취합 | 월 단위 사후 파악. 에이전트가 자율 호출하는 서드파티 API 비용은 아예 안 잡힘 |
| **프로바이더 하드 캡 + 빌링 알림** | OpenAI/Anthropic 지출 한도, 클라우드 예산 알림 설정 | 한도 도달 시 서비스가 통째로 죽거나(hard cap), 알림은 이미 돈이 나간 뒤(soft alert). 중간이 없음 |
| **아무것도 안 함** | 지출 규모가 아직 작아 무관심 | 문제 심각도 임계점 미달 세그먼트 — ICP에서 제외해야 할 신호 |

Status quo를 이기려면 "스크립트보다 편하다"가 아니라, status quo로는 **구조적으로 불가능한 것** — 에이전트의 자율 호출을 실행 전에 통제, LLM + 외부 API 통합 실시간 비용 뷰 — 을 보여줘야 한다.

## 4. 시사점 (해석 — 검증 필요)

1. **SatGate의 존재는 양날이다.** 문제·솔루션이 거의 동일한 팀이 이미 있다 = 문제가 실존한다는 강한 검증 신호. 동시에 프리론칭 단계라 아무도 시장을 못 잡았다 = 차별화와 속도가 관건.
2. **"사후 관측 vs 실행 전 통제"가 시장의 단층선이다.** 관측성·FinOps 진영(다수, 성숙)은 사후에, SatGate·Cycles·LiteLLM 세션 예산(소수, 초기)은 실행 전에 선다. 우리 문제 정의("추적과 통제가 어렵다")는 후자에 속한다.
3. **Cycles는 경쟁자이기 전에 표준 후보다.** 프로토콜이 표준이 되면 "누가 그 위의 제품을 잘 만드느냐" 게임이 된다.
4. **인터뷰에서 확인할 것:** 고객이 실제로 status quo(로그 뒤지기·스크립트)에서 얼마나 고통받는지, 실행 전 통제에 돈을 낼 의사가 있는지, 이미 LiteLLM류 게이트웨이를 쓰는 곳도 이 문제를 여전히 겪는지.
