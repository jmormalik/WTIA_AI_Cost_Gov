# 리스크 분석 및 대책 (2026-07)

> 작성 배경: 현재 문서 전체(`Problem/`, `ICP/`, `Buyer-Persona/`, `Competitor-Analysis/`, `Solution/`, `Market/`)를 검토해 아이디어의 구조적 약점을 찾고 대책을 정리한 문서.
>
> ⚠️ 이 문서의 성격: **비판적 검토 결과이지 사형 선고가 아니다.** 각 약점에 대책을 함께 적었다. 다만 대책 대부분은 **인터뷰로 검증되기 전까지는 또 다른 가설**임을 명시한다.
>
> 상태: 인터뷰 0건 시점에 작성 — 아래 판단들은 문서·웹 조사 기반 추론이다.

---

## 요약 (Executive Summary)

가장 치명적인 것 셋만 고르면:

1. **플랫폼 흡수** — Anthropic이 이미 2026-07-02에 모델별 권한·비용 대시보드·Admin API를 출시했다. 프로바이더가 네이티브로 흡수하면 카테고리가 사라진다.
2. **신뢰 ↔ 데이터의 순환 종속** — moat를 만들려면 요청 경로 데이터가 필요한데, 요청 경로에 들어가려면 신뢰가 필요하고, 초기 팀에겐 그 신뢰가 없다. 구조적 데드락.
3. **LiteLLM이 핵심을 이미 커버** — 남은 쐐기(툴·외부 API 비용)가 지출의 10~20%뿐이라면 "10배 개선"이 성립하지 않는다.

가장 강력한 단일 대책: **"요청 경로 밖에서 시작해, 신뢰를 얻은 뒤 경로 안으로 들어간다."** (아래 대책 A)

# 1부 · 치명적 약점

## 약점 1. 플랫폼 흡수 리스크 (치명도 ★★★★★)

**내용**
모델 프로바이더·클라우드가 비용 통제를 네이티브 기능으로 넣으면 우리 카테고리 자체가 증발한다.

**근거 — 이미 시작됐다**
- Anthropic (2026-07-02): 모델 수준 권한 설정, 그룹·사용자별 비용 분석 대시보드, 한도 증액 워크플로우 Admin API 출시 (`Competitor-Analysis` 참조)
- AWS Bedrock·Azure도 같은 방향의 거버넌스 기능 확장 중

**왜 치명적인가**
프로바이더는 bill shock를 없앨 강력한 인센티브가 있다 — 청구 분쟁 감소, 이탈 방지, 엔터프라이즈 판매 촉진. 그리고 자사 API 내부에서는 우리보다 정확하게 통제할 수 있다. 투자자의 고전적 거절 사유인 **"feature, not a company"**에 정면으로 걸린다.

**반드시 받을 질문:** *"왜 OpenAI나 Anthropic이 이걸 직접 하지 않을까요?"* — 현재 우리 문서에 답이 없다.

---

## 약점 2. 신뢰 ↔ 데이터 순환 종속 (치명도 ★★★★★)

**구조**

```
moat를 만들려면        → 트래픽 데이터가 쌓여야 한다
데이터가 쌓이려면      → 요청 경로 안에 있어야 한다
요청 경로에 들어가려면 → 프로덕션 크리티컬 패스를 맡길 신뢰가 필요하다
그런데                 → 초기 팀에게 그 신뢰는 없다  ← 순환이 닫힘
```

**근거**
- `Solution/moat-후보.md`의 moat 후보 대부분(루프 패턴 감지, 벤치마크, 시뮬레이션 정확도)이 **트래픽 데이터 축적**에 의존
- `Buyer-Persona` 두 문서 **모두** 최대 장벽으로 "호출 경로에 뭘 끼우는 건 무섭다 — 레이턴시·SPOF"를 기록
- 추가로 프롬프트에는 고객 PII·영업비밀이 흐른다 → 보안 심사 대상

**탈출 시도의 함정**
self-hosted/OSS로 신뢰를 얻으면 → 데이터가 우리에게 안 모임 → moat 소멸 → 결국 LiteLLM과 같아진다.

---

## 약점 3. LiteLLM 커버리지 — 남은 쐐기가 10배인가 (치명도 ★★★★☆)

**LiteLLM이 이미 제공하는 것** (무료·OSS·성숙·100+ 프로바이더)
- 세션별 예산 상한 (`max_budget_per_session`)
- 반복 상한 (`max_iterations`)
- 예산 초과 시 요청 차단 (`ExceededTokenBudget`)
- 예산 폴백 (저가 모델 전환)
- 키/유저/팀/고객/세션 단위 귀속

**우리에게 남은 두 베팅과 각각의 약점**

| 베팅 | 약점 |
|---|---|
| (a) 툴·외부 API까지 총비용 커버 | 에이전트 지출의 **대부분은 LLM 토큰**이다. 외부 API가 10~20%라면 이는 주변부 개선이지 10배가 아니다 → **인터뷰 최우선 검증 대상** |
| (b) "기능 존재 ≠ 조직이 통제 가능" | 이건 제품 문제가 아니라 **교육·서비스 문제**일 가능성이 높다. UX 개선만으로는 6개월이면 복제된다 |

수업에서 배운 원칙: 전환을 일으키려면 **10배 개선**이 필요하다. 위 둘이 10배인지 아직 아무도 모른다.

---

## 약점 4. ICP와 페인의 어긋남 + 벤처 규모 미달 (치명도 ★★★★☆)

**모순 구조**
- 사고 사례 규모: **$47K ~ $500M** → 엔터프라이즈 영역
- 우리 비치헤드: **ACV $8k 한국 AI 스타트업**
- 우리 크레딧 트리거 가설: 그 스타트업들은 **크레딧 때문에 아직 안 아프다**
- 게다가 한국 초기 스타트업은 인프라 거버넌스에 **가장 돈을 안 쓰는 집단** — 무료 OSS(LiteLLM)를 고를 유인이 압도적

> 아픈 사람에겐 못 팔고(엔터프라이즈, 6~18개월 조달), 팔 수 있는 사람은 안 아프고(크레딧 중), 아파도 예산이 없다.

**규모 문제**
현재 산정: **LAM $80,000 · SOM ~$180,000/년.** 수업에서 배웠듯 VC는 수십억 달러 시장을 본다. 이 숫자를 그대로 피치덱 시장 슬라이드에 올리면 그 자리에서 탈락한다.

---

## 약점 5. 에피소드성 긴급성 → 구조적 이탈(Churn) (치명도 ★★★☆☆)

비용 거버넌스는 **사고 직후에만 뜨거운** 제품이다.

- 트리거(사고·크레딧 소진)가 지나가면 "캡 걸었으니 해결됐다"고 느낀다
- 성장기에는 출시가 우선, 비용 통제는 후순위
- 예산 삭감 시 **가장 먼저 잘리는 라인아이템**

즉 진통제였다가 통증이 가시면 비타민이 된다. 갱신 시점에 이탈이 몰릴 구조다.

---

## 약점 6. 두 진영 사이 무인지대 (치명도 ★★★☆☆)

| 진영 | 플레이어 | 우리와의 관계 |
|---|---|---|
| 강제(Enforcement) 층 | SatGate, Cycles, LiteLLM | SatGate는 문제 정의·솔루션 형태 동일 + macaroon 위임 + MCP 툴 콜 귀속 + 서명 감사 + **특허 출원 중** |
| 재무(Finance) 층 | CloudZero, Finout, Vantage | 유닛 이코노믹스·COGS는 **이미 이들의 본업** |

우리의 C군 베팅(유닛 이코노믹스·시뮬레이션·벤치마크)은 결국 **자금력 있는 FinOps 진영과 정면 승부**다. "아무도 없는 중간"이라 믿는 자리가 실은 **양쪽 모두에게 쉬운 확장 영역**일 수 있다.

---

## 약점 7. 가격 모델의 구조적 모순 (치명도 ★★☆☆☆)

지출 연동 과금(% of spend)은 **성공할수록 고객 지출이 줄어 우리 매출도 줄어드는** 절감형 제품의 고전적 함정이다. "차단한 지출(avoided spend)" 기준 과금은 측정·감사가 사실상 불가능하다.

---

## 약점 8. 검증 0 + 높은 인프라 난이도 (치명도 ★★★☆☆)

- **인터뷰 0건.** 모든 문서에 "⚠️ 검증되지 않은 가설" 표기 — 정직하지만, 곧 피치를 해야 한다
- 프로덕션 크리티컬 패스의 **저지연·고가용 프록시**는 난이도가 높다: 스트리밍 지원, 페일오버, 멀티 프로바이더 드리프트, 레이턴시 예산 관리. 초기 소규모 팀에게 무거운 과제

# 2부 · 대책

## 구조적 대응 (4가지)

### 대책 A. 요청 경로 밖에서 시작해, 경로 안으로 들어간다 ★ 최우선

**지금 프록시를 만들지 않는다.** 대신 **기존 로그·게이트웨이 익스포트를 받아 분석하는 리포트 제품**부터 만든다.

- 고객은 LiteLLM/Langfuse/프로바이더 로그를 내보내기만 하면 됨 → **크리티컬 패스 리스크 0, 신뢰 장벽 0, 보안 심사 최소화**
- 산출물: **① 사고 재구성 리포트**("지난달 무엇이 왜 얼마를 썼는가") **② 고객 1명당 AI 원가·마진 뷰**
- 이는 우리가 **1순위 페인으로 꼽은 "설명 불가"(심각도 9점)**를 정면으로 해결한다
- 리포트가 사랑받으면, 그때 **"실시간으로 막아드릴까요?"**로 경로 진입을 요청할 권리를 얻는다

**이 하나로 동시에 해소되는 약점: 2(신뢰↔데이터), 3(LiteLLM 대체 불필요), 8(인프라 난이도).**

### 대책 B. LiteLLM과 싸우지 말고 그 위에 올라탄다

게이트웨이를 **대체**하지 말고 게이트웨이의 **정책·회계 레이어**로 포지셔닝한다. LiteLLM 플러그인/미들웨어로 붙으면:

- "잘 쓰는 게이트웨이를 왜 바꾸나?" 반론 소멸
- 기존 채택 기반 위에 올라탐
- SPOF·레이턴시 반론 대폭 감소

> **포지셔닝 변경: "AI 게이트웨이" → "에이전트 지출 회계·정책 레이어"**

### 대책 C. 플랫폼 흡수에 대한 답 = 중립성 + 감사 독립성

프로바이더는 **자기 지출만** 통제한다. 고객의 실제 고통은 OpenAI + Anthropic + 툴 + 인프라의 **합계**다.

그리고 결정적으로 — **벤더가 자기 청구서를 감사하는 것은 증거 능력이 없다.** 재무·이사회·외부 감사에 제출 가능한 것은 **독립적인 제3자 기록**뿐이다.

> 이것이 "왜 OpenAI가 안 하나?"에 대한 유일하게 방어 가능한 답이다: **크로스 벤더 중립 + 독립 감사 증적.**

### 대책 D. 프라이버시 보존 텔레메트리 — 데이터 없이 moat 만들기

처음부터 **메타데이터만** 수집하도록 설계한다.

- 수집: 토큰 수, 모델명, 툴 이름, 지연시간, 비용, 호출 패턴
- **비수집: 프롬프트·응답 본문 (절대)**

→ 보안 심사를 통과하면서도 **크로스 고객 벤치마크·패턴 감지**가 가능해진다. 순환 종속(약점 2)의 유일한 탈출구.

> ⚠️ 이것은 **지금 내려야 하는 아키텍처 결정**이다. 나중에는 되돌리기 어렵다.

| 7 · 가격 | 절감액 연동 금지. **좌석/워크로드 기반 정액 + 사용량 티어**. 파는 것은 절감이 아니라 **통제와 설명 가능성** |

# 3부 · 즉시 실행 (2주)

## 1. 인터뷰 10~15건 (최우선)

가이드·후보 리스트·아웃리치 메일이 이미 준비되어 있다(`Interview/`). **실행만 남았다.**

**이번 인터뷰의 생사 질문 2개:**

- **Q. 툴·외부 API 비용이 전체 AI 지출의 몇 %인가?** → 약점 3의 생사. 10~20%면 쐐기를 바꿔야 한다
- **Q. 요청 경로에 외부 프록시를 넣을 의향이 있는가? 조건은?** → 약점 2의 생사. "절대 불가"가 다수면 대책 A가 필수가 된다

부수 확인: 크레딧 소진 시점 즉답 가능 여부(Q3), 사고 시 원인 추적 과정(Q8), 지불 이력(Q14).

## 2. 경로 밖 리포트 MVP

로그를 받아 **사고 재구성 + 고객당 원가 리포트**를 산출. 디자인 파트너 3곳 확보 목표.

## 3. 아키텍처 결정 문서화

**메타데이터 온리 텔레메트리**(대책 D)를 `Solution/`에 결정 기록으로 남긴다.

가장 중요한 한 가지를 고르면 **대책 A(경로 밖에서 시작)**이다. 신뢰 장벽·인프라 난이도·LiteLLM 경쟁을 동시에 우회하면서, 우리가 1순위로 꼽은 "설명 불가" 페인을 정면으로 친다.

그리고 위 분석과 대책 전부는 **문서 기반 추론**이다. **인터뷰 10건이 이 문서 전체보다 가치 있다.** 2주 안에 실행하는 것이 유일한 다음 단계다.

## 다음 단계

1. 이 문서를 팀에서 함께 검토 — 특히 약점 3·4에 대한 반론이 있는지
2. 인터뷰 실행 (생사 질문 2개 포함)
3. 인터뷰 결과로 이 문서를 개정 — 어떤 약점이 실제였고 어떤 게 기우였는지 기록
