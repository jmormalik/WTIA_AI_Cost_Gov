# 문제 정의

## 핵심 문제 (가설)

AI를 사용해 고객 답변이나 결과를 생성할 때, AI 모델(에이전트)이 스스로 툴 콜링이나 API 호출을 수행한다. 어떤 툴을 언제, 몇 번 호출할지를 AI 모델이 스스로 판단·평가하기 때문에, 이 호출들을 **추적하고 통제하기 어렵다**.

> ⚠️ 상태: 검증되지 않은 가설. 고객 인터뷰로 확인된 사실 아님.

## 문제 풀어쓰기

- 전통적인 API 사용은 개발자가 코드에 호출 시점·횟수를 명시하므로 예측과 통제가 가능하다.
- 반면 에이전틱 AI는 하나의 사용자 요청을 처리하는 과정에서 모델이 자율적으로 여러 번의 툴 콜/외부 API 호출을 결정한다.
- 호출 여부·횟수·대상이 모델의 자체 판단에 달려 있어:
  - **추적(관측)의 어려움** — 어떤 요청이 어떤 호출들을 유발했는지, 비용이 어디서 발생했는지 파악하기 어렵다.
  - **통제(거버넌스)의 어려움** — 호출 횟수·비용에 상한을 걸거나, 불필요한 호출을 사전에 막기 어렵다.

## 검증해야 할 하위 가설

> 경쟁 환경 조사([../Competitor-Analysis/경쟁사-분석.md](../Competitor-Analysis/경쟁사-분석.md), 2026-07)를 반영해 구체화함.

1. 기업들이 실제로 에이전트의 자율적 툴 콜/API 호출로 인한 비용 예측 불가·급증을 경험하고 있다.
2. 기존 LLM Gateway(LiteLLM, Portkey 등)는 **LLM 호출 비용 추적·예산 기능은 성숙**했지만, 에이전트가 자율적으로 호출하는 **툴/외부 API까지 포함한 총비용의 추적·귀속·통제는 미성숙**하다 (MCP 거버넌스는 대부분 기술 미리보기 수준).
3. 예산 상한 같은 **기능이 존재하는 것**과 조직이 실제로 **비용을 예측·귀속·통제할 수 있는 것**은 다르다 — 게이트웨이를 도입한 기업도 여전히 이 문제를 겪는다.
4. 이 문제의 심각도·빈도가 비용을 지불하고 해결할 만한 수준이다.

⚠️ 주의: 단순히 "기존 도구로 통제 불가"라는 가설은 조사 결과 성립하지 않음 (LiteLLM은 이미 에이전트 세션별 예산·반복 상한 제공). 위 2·3번처럼 좁힌 형태로 인터뷰에서 검증해야 한다.

## Why Now

타이밍 논거와 실제 사고 사례([why-now.md](why-now.md)): 에이전트의 프로덕션 전환으로 토큰 소모가 챗봇의 ~50배가 됐고, 호출 결정 주체가 사람에서 모델로 넘어갔으며, 사고가 이미 전 규모에서 발생 중($4K 개발자 1인 ~ $500M 대기업). 기업 73%가 AI 비용 예상 초과를 보고.
